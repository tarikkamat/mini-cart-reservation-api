# API Reference

All endpoints under `/api`, JSON in / JSON out, throttled to **60
req/min/IP**. Every response carries an `X-Request-Id` header (echoed if
supplied, otherwise generated) shared with all log lines for correlation.

For the idempotency contract see [CONCURRENCY.md](CONCURRENCY.md#idempotency--redis-as-single-source-of-truth).

## `GET /api/health`

```bash
curl http://localhost:8000/api/health
# {"status":"ok","db":"ok","redis":"ok"}
```

## `GET /api/products`

Query params: `search`, `is_active` (default `true`), `sort`
(`newest` / `price_asc` / `price_desc`), `per_page` (≤ 100), `page`.

```bash
curl "http://localhost:8000/api/products?search=mug&sort=price_asc&per_page=10"
```

Response includes `price: { amount, currency }` and a real-time
`available_stock` that always equals `quantity - reserved_quantity`.

**Performance contract:** ≤ 4 SQL queries regardless of result count,
enforced by an N+1 test that asserts the query log size.

## `POST /api/cart/quote`

Server-priced cart calculation. The client sends only `product_id` +
`quantity`; any client-supplied `unit_price` is ignored.

```bash
curl -X POST http://localhost:8000/api/cart/quote \
  -H "Content-Type: application/json" \
  -d '{
    "customer_email": "jane@example.com",
    "items": [
      {"product_id": 1, "quantity": 2},
      {"product_id": 5, "quantity": 1}
    ]
  }'
```

## `POST /api/reservations`

Creates an active reservation, atomically reserving stock for ~15 minutes
(see `RESERVATION_TTL_MINUTES`).

```bash
curl -X POST http://localhost:8000/api/reservations \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: 7f3e9d52-1c6a-4ba3-9c45-9a2c1d5b2c11" \
  -d '{
    "customer_email": "jane@example.com",
    "items": [{"product_id": 1, "quantity": 2}]
  }'
```

The `Idempotency-Key` header is **strongly recommended**:

- Same key + same body within 24h → returns the original 201 response with
  `Idempotent-Replay: true`. No second reservation is created.
- Same key + different body → `422 IDEMPOTENCY_CONFLICT`.

## `POST /api/reservations/{id}/release`

Idempotent by nature. Calling on an already-released, expired, or committed
reservation returns the current state with HTTP 200, no double-restore of
stock.

```bash
curl -X POST http://localhost:8000/api/reservations/7f3e9d52-.../release
```

## Interactive API documentation

OpenAPI 3.1 docs are auto-generated by
[Scramble](https://github.com/dedoc/scramble) from the routes,
FormRequests, and API Resources — no annotations to maintain:

- UI: <http://localhost/docs/api>
- Spec: <http://localhost/docs/api.json>

Snapshot the spec to `docs/openapi.json` for releases or downstream tooling:

```bash
make docs-export
```

The Scramble UI is gated to `app()->environment('local')` by default so it
does not ship to production.

## Error format

All 4xx errors share this shape:

```json
{
  "error": {
    "code": "INSUFFICIENT_STOCK",
    "message": "Requested quantity exceeds available stock.",
    "details": [{ "product_id": 5, "requested": 10, "available": 3 }]
  }
}
```

| Code | HTTP | Trigger |
|---|---|---|
| `VALIDATION_FAILED` | 422 | Schema / type errors |
| `PRODUCT_NOT_FOUND` | 422 | `product_id` doesn't exist |
| `PRODUCT_INACTIVE` | 422 | Product `is_active = false` |
| `MIXED_CURRENCY` | 422 | Cart spans multiple currencies |
| `IDEMPOTENCY_CONFLICT` | 422 | Same key, different body |
| `INSUFFICIENT_STOCK` | 409 | Requested qty > available |
| `NOT_FOUND` | 404 | Reservation / route doesn't exist |
